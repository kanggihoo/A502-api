# GitLab 추적 대상 설정 — 사용자 흐름(User Flow)

> **목적**: OAuth2 로그인 이후, 사용자가 "어느 GitLab 그룹/프로젝트를 통합 워크스페이스에서 추적할지" 결정하는 온보딩 흐름을 정의한다.
>
> **관련 가설**: `AGENTS.md` 우선 검증 가설 2(통합 알림과 대시보드), 3(프로젝트 운영 지원)
>
> **상태**: 설계안(POC 단계). 미검증 항목은 본문 말미에 명시.

## 1. 배경과 문제

OAuth2 로그인으로 "사용자가 누구인지"는 알 수 있지만, **"어떤 리소스를 추적할지"는 별개 문제**다.

SSAFY 구조상 한 사용자는 여러 학기·여러 단계의 프로젝트에 소속된다:

- `instruction/15th/skeleton-projects/...` — 1학기 교육용 skeleton
- `s15-webmobile1-sub1/S15P11A502` — 2학기 팀 프로젝트
- `s15/a09/miniproject/01.java/...` — 2학기 미니프로젝트 (단계별)

`GET /projects?membership=true`는 접근 가능한 모든 프로젝트를 반환하므로, 사용자는 수십 개의 후보를 받게 된다. 이 중 "지금 의미 있는 것"만 자동으로 고를 수는 없다 — 사용자 의도를 가정하면 안 된다(`AGENTS.md` 설계 원칙).

따라서 사용자가 **최소한의 입력**으로 추적 대상을 지정할 수 있어야 한다.

## 2. 설계 원칙

- **사용자 입력은 최소화**하되, 완전 자동화는 지양한다(오추론 = 알림 폭탄).
- SSAFY 그룹 path의 **명명 규칙을 UI 분류에만 활용**한다(자동 선택 아님).
- 추적 대상은 **그룹 단위로 묶어 선택**하고, 하위 프로젝트는 자동으로 확보한다.
- 민감 정보(access_token, refresh_token)는 저장소에 커밋하지 않는다.
- 미검증 동작(권한, 페이지네이션, 하위 그룹 포함 여부)은 사실로 가정하지 않는다.

## 3. 사용자 흐름 (User Flow)

```
┌─────────────────────────────────────────────────────────────┐
│ 1. OAuth2 로그인                                              │
│    GitLab Authorize 화면 → access_token 발급                  │
└──────────────────────────────┬──────────────────────────────┘
                               ▼
┌─────────────────────────────────────────────────────────────┐
│ 2. 사용자 식별                                                │
│    GET /user → user_id, username 확보                          │
└──────────────────────────────┬──────────────────────────────┘
                               ▼
┌─────────────────────────────────────────────────────────────┐
│ 3. 접근 가능 그룹 수집                                         │
│    GET /groups?membership=true (페이지네이션 전체 순회)        │
└──────────────────────────────┬──────────────────────────────┘
                               ▼
┌─────────────────────────────────────────────────────────────┐
│ 4. path 기반 UI 분류                                          │
│    - 📚 교육 자료      (instruction/*)                        │
│    - 🚀 이번 기수 프로젝트 (s<N>-*)                           │
│    - 📂 기타                                                  │
└──────────────────────────────┬──────────────────────────────┘
                               ▼
┌─────────────────────────────────────────────────────────────┐
│ 5. 사용자: 추적할 그룹 선택 (체크박스, 복수 가능)              │
└──────────────────────────────┬──────────────────────────────┘
                               ▼
┌─────────────────────────────────────────────────────────────┐
│ 6. 하위 프로젝트 자동 확보                                     │
│    GET /groups/<선택 그룹 ID>/projects                         │
│    (include_subgroups=true 검증 필요)                          │
└──────────────────────────────┬──────────────────────────────┘
                               ▼
┌─────────────────────────────────────────────────────────────┐
│ 7. 사용자: 추적 프로젝트 최종 필터 (기본 = 전체)               │
└──────────────────────────────┬──────────────────────────────┘
                               ▼
┌─────────────────────────────────────────────────────────────┐
│ 8. 추적 설정 저장                                              │
│    { user_id, cohort, groups[], projects[] }                  │
└─────────────────────────────────────────────────────────────┘
```

## 4. 단계별 상세

### 4.1 OAuth2 로그인

- Authorization Code with PKCE 흐름 사용 (공식 문서 권장).
- scope: `api` (webhook 생성까지 POC 범위에 포함).
- Redirect URI: HTTPS 필수 (`lab.ssafy.com` = 프로덕션).
- 참고: `api-docs/gitlab_rest_defuddle_markdown/oauth2.md`

### 4.2 사용자 식별

```
GET {{base_url}}/api/v4/user
Authorization: Bearer {{access_token}}
```

확보 필드: `id`, `username`, `name`, `state`.

> **미검증**: `/user` 응답에 기수(ssafy 15기 등) 정보가 있는지 확인 필요. 없으면 온보딩 시 사용자 입력 또는 path 접두(`s15`)로 추론.

### 4.3 접근 가능 그룹 수집

```
GET {{base_url}}/api/v4/groups?membership=true&per_page=100&page=1
GET ...&page=2
... (X-Next-Page 헤더가 빈 값일 때까지)
```

> **미검증**: 총 그룹 개수, 페이지네이션 동작(`X-Next-Page` / `X-Total-Pages` 헤더). POC에서 전체 순회 후 총 개수 기록.

### 4.4 path 기반 분류

분류 규칙 (초안):

```python
def classify_group(full_path: str, cohort: str) -> str:
    if full_path.startswith("instruction"):
        return "education"
    if full_path.startswith(cohort):        # 예: cohort = "s15"
        return "cohort_project"
    return "other"
```

`cohort`는 하드코딩하지 않는다 — `/user` 추론 실패 시 온보딩에서 입력받는다.

> **미검증**: SSAFY 명명 규칙 일관성. 다른 사용자/기수에서도 `s<N>` 접두가 동일한지. POC에서 여러 사용자 데이터 수집 필요.

### 4.5 추적 그룹 선택

UI에서 분류된 그룹 목록을 체크박스로 제공.

- 기본 체크: `cohort_project` 분류 (사용자가 1회 확인 후 변경)
- 교육/기타 분류는 접어두고 기본 펼침 없음

> 사용자가 의도를 가정하지 않도록, 자동 선택은 "추천"으로만 표시하고 저장 전 확인을 강제한다.

### 4.6 하위 프로젝트 자동 확보

```
GET {{base_url}}/api/v4/groups/<group_id>/projects?include_subgroups=true&per_page=100
```

> **미검증**: `include_subgroups=true` 동작 여부 — 하위 그룹의 프로젝트까지 한 번에 가져오는지. 검증 전에는 상위 그룹만 조회 후, 하위 그룹도 사용자가 별도 선택해야 할 수 있음.

### 4.7 추적 프로젝트 최종 필터

- 기본값: 그룹에서 확보한 프로젝트 전부 선택
- 사용자가 일부 해제 가능 (예: skeleton, archive 제외)

### 4.8 추적 설정 저장

저장 형태:

```json
{
  "user_id": 30128,
  "username": "11kkh19",
  "cohort": "s15",
  "tracking": {
    "groups": [
      {
        "id": 1250XXX,
        "full_path": "s15-webmobile1-sub1",
        "web_url": "https://lab.ssafy.com/groups/s15-webmobile1-sub1"
      }
    ],
    "projects": [
      {
        "id": 1373907,
        "path_with_namespace": "s15-webmobile1-sub1/S15P11A502",
        "web_url": "https://lab.ssafy.com/s15-webmobile1-sub1/S15P11A502"
      }
    ]
  },
  "created_at": "2026-07-19T17:30:00+09:00",
  "updated_at": "2026-07-19T17:30:00+09:00"
}
```

**민감 정보 분리**: `access_token`, `refresh_token`은 별도 보안 저장소. 본 설정 파일에는 커밋 금지.

## 5. 사용자 입력 요약

| 단계 | 사용자 행동 | 필수 여부 |
|------|-------------|-----------|
| 로그인 | GitLab 인증 | 필수 |
| 기수 추론 실패 시 | 기수 입력 (예: `s15`) | 조건부 |
| 추적 그룹 선택 | 체크박스 확인/변경 | 필수 (기본값 제공) |
| 추적 프로젝트 필터 | 일부 해제 (선택) | 선택 |

목표: **실제 입력 = 체크박스 확인 1회**. 나머지는 자동 확보.

## 6. 재설정 흐름 (변경 시)

- 학기 시작/팀 변경 시 추적 대상 업데이트 필요
- 동일한 흐름 재진입 → 기존 설정 불러오기 → diff 표시 → 저장
- 이미 수집한 알림 히스토리는 유지

## 7. 예외 상황

| 상황 | 동작 |
|------|------|
| access_token 만료 | refresh_token으로 자동 갱신 (rotate 방식 — 이전 토큰 무효화됨, 재사용 불가) |
| refresh_token 만료 | 재로그인 요청 |
| 선택한 그룹/프로젝트 접근 권한 상실 | 해당 항목 비활성화 + 사용자 알림, 추적에서 자동 제외 |
| 그룹 페이지네이션 실패 | 부분 목록으로 진행 + "전체 불러오기 재시도" 버튼 |
| path 패턴 미분류 그룹 | `other` 분류로 표시, 사용자가 직접 선택 가능 |

## 8. 미검증 항목 (POC에서 확인 필요)

> `AGENTS.md` 원칙 — 확인하지 않은 동작을 사실로 가정하지 않는다.

1. **그룹 페이지네이션** — 총 개수, `X-Next-Page` 동작
2. **path 명명 규칙 일관성** — 다른 기수/사용자에서도 동일한가
3. **`instruction` 그룹 가시성** — 모든 사용자 노출 vs 일부
4. **`include_subgroups=true` 동작** — 하위 그룹 프로젝트 자동 확보 가능성
5. **`/user` 응답의 기수 정보** — 자동 추론 가능 여부
6. **`lab.ssafy.com` OAuth 앱 등록 정책** — redirect_uri HTTPS 강제 여부
7. **scope 축소 가능 여부** — `api` 대신 `read_api` + webhook 전용 권한 분리

## 9. 후속 결정 사항

- 추적 대상 변경 시 알림 히스토리 보존 정책
- 다중 팀 소속 사용자의 추적 그룹 상한
- 사용자별 추적 데이터 저장 위치 (DB schema)
- 기수 자동 추론 규칙의 검증 데이터 출처
