이 제품은 여러 협업 도구를 다시 만드는 대시보드가 아니라, 팀이 지금 해야 할 일·진행 흐름·일정을 한곳에서 판단하고 행동하게 하는 프로젝트 운영 워크스페이스다.

## 팀 프로젝트 운영 워크스페이스 기능 Spec

### 제품 목적

SSAFY 팀 프로젝트에서 사용하는 GitLab, Jira, Notion, Mattermost, Jenkins의 정보를 한 화면에 모은다. 사용자는 각 도구를 오가며 상태를 찾지 않고, 필요한 원문으로 즉시 이동한다.

핵심 가치는 다음 세 가지다.

- **오늘의 행동:** 지금 확인하거나 처리해야 하는 일을 빠르게 찾는다.
- **팀의 흐름:** 이슈부터 MR, Pipeline, 배포, 문서까지 기능의 진행 상황을 본다.
- **팀의 기록:** 스크럼·일정·주간 기록을 남기고 나중에 다시 찾는다.

### 목표와 비목표

| 구분 | 내용 |
| --- | --- |
| 목표 | 분산된 프로젝트 상태를 행동 가능한 정보로 요약한다. |
| 목표 | 팀 스크럼과 중앙 일정을 일관된 흐름으로 운영한다. |
| 목표 | 원문 데이터와 권한은 각 도구에 남기고, 서비스는 링크·상태·요약을 제공한다. |
| 비목표 | GitLab, Jira, Notion의 모든 기능을 서비스 안에서 재구현하지 않는다. |
| 비목표 | 커밋 수 기반 개인 랭킹이나 경쟁 기능을 기본 기능으로 제공하지 않는다. |
| 비목표 | 임의의 Notion 페이지·DB를 자동 수집하거나 전체 워크스페이스를 감시하지 않는다. |

### 사용자와 역할

| 역할 | 책임 |
| --- | --- |
| 팀장 | 팀 워크스페이스 생성, 연동 연결, 핵심 일정 규칙 설정 |
| 팀원 | 개인 작업·리뷰·스크럼·일정 확인과 원문 데이터 관리 |
| 일정 담당자 | 중앙 일정 생성·수정, 발표·제출·회의 일정 최신화 |
| 인프라 담당자 | GitLab webhook, Jenkins, 배포·Pipeline 연동 검증 |

팀장은 모든 일정을 직접 관리하지 않는다. 일정마다 담당자를 지정하고, 담당자가 해당 일정의 최신성을 책임진다.

### 핵심 화면과 기능

#### 오늘의 팀 홈

홈은 전체 도구의 모든 데이터를 나열하지 않는다. 다음 정보만 우선 표시한다.

- 전체 이슈 수와 완료·진행·미할당·마감 경과 수
- 내게 할당된 Jira 이슈와 마감 임박 항목
- 리뷰가 필요한 MR과 오래 열린 MR
- 실패·진행 중인 Pipeline 및 최근 배포 상태
- 오늘 스크럼 문서와 다가오는 핵심 일정
- 팀이 해결해야 할 위험 신호

각 항목은 GitLab, Jira, Notion 등 원문 URL을 제공한다. 서비스 안에서 상세 편집을 강요하지 않는다.

#### 작업 흐름

기능 또는 작업의 연결 관계를 보여 준다.

`Jira 이슈 → GitLab 브랜치/MR → Pipeline → 배포 → Notion 기록`

이 화면의 목적은 코드 그래프 자체가 아니라, 작업이 어느 단계에서 멈췄는지 확인하는 것이다. Git graph는 브랜치·병합 문제를 확인하는 상세 보조 화면으로만 둔다.

#### 스크럼 허브

- 팀 공통 기본 템플릿 제공: 어제 한 일, 오늘 할 일, 막힌 점, 도움 요청, 결정 사항
- 팀별 템플릿 수정과 필수 항목 설정
- 날짜별 스크럼 문서 생성
- 사용자가 직접 작성
- 이전 미완료 항목, 관련 Jira 이슈·GitLab MR을 참조 패널에 제공
- 완료된 문서는 Notion에 저장하고 서비스에는 문서 ID·제목·링크·상태만 보관

초기 범위에는 음성 녹음, Mattermost 채팅 자동 수집, AI 초안 작성이 없다. AI 문장 다듬기와 주간 회고 요약은 후속 기능이다.

#### 중앙 캘린더

중앙 캘린더는 팀 일정을 보는 기준 화면이다. 일정 원본은 하나만 유지한다.

| 일정 유형 | 원본 | 중앙 캘린더 동작 | Mattermost 동작 |
| --- | --- | --- | --- |
| 발표, 문서 제출, 회의, 배포 시간 | 서비스 중앙 캘린더 | 생성·수정·담당자·알림 관리 | 변경·리마인드 공지 |
| Jira 이슈 마감일, 스프린트 | Jira | 가져와 표시, Jira 링크 제공 | 필요 시 리마인드 |
| Notion 문서/간단 일정 | 팀이 선택한 Notion 일정 DB | 가져와 표시, Notion 링크 제공 | 필요 시 리마인드 |
| GitLab 마일스톤, 릴리스, 배포 | GitLab | 가져와 표시, GitLab 링크 제공 | 필요 시 리마인드 |

중앙에서 만든 일정은 Notion의 팀 일정 DB에 단방향으로 기록한다. Jira와 Notion에서 이미 관리되는 외부 일정은 초기에는 읽기 전용으로 가져온다. 외부 일정의 날짜를 서비스에서 바꾸려면 원문 도구로 이동한다.

#### Notion 일정 연결

Notion 연동은 필수가 아니다. 팀은 다음 중 하나를 선택한다.

1. **Notion 없이 시작:** 서비스 중앙 캘린더만 사용한다.
2. **기존 일정 선택:** Connection이 접근 가능한 Notion 일정 DB를 이름과 미리보기로 선택한다.
3. **팀 일정 DB 만들기:** 사용자가 지정한 상위 Notion 페이지 아래에 서비스가 팀 일정 DB를 만든다.

사용자에게 Notion 블록 ID나 데이터 소스 ID를 입력시키지 않는다. 사용자가 Notion 화면에서 Connection 공유를 한 뒤, 서비스가 접근 가능한 목록을 보여 준다.

팀당 쓰기 대상 Notion 일정 DB는 기본 1개다. 다른 Notion 일정 DB는 후속 단계에서 읽기 전용 외부 일정으로 추가할 수 있다. 사용자가 새 Notion 캘린더를 임의로 만들면 자동 추적하지 않으며, 사용자가 연결 대상으로 선택할 때만 포함한다.

#### 기간 통계와 기록

통계는 개인 순위가 아니라 팀 진행·병목·회고를 위한 정보다.

- 기간 전환: 오늘, 7일, 30일, 프로젝트 전체
- 이슈: 생성·완료·진행·마감 경과 추이
- MR: 생성·병합·리뷰 대기·평균 대기 시간
- CI/CD: Pipeline 성공률·실패 횟수·실행 시간·배포 횟수
- 기록: 스크럼 작성률, 주간 완료 작업, 주요 결정·배포 이력

권장 시각화는 완료 비율, 일별 완료 추이, MR 생성/병합 추이, Pipeline 성공/실패 비율, 일정 타임라인이다. 개인 기여 통계는 본인 상세 화면에서만 제공하며 공개 비교를 기본값으로 하지 않는다.

#### 알림함

알림은 도구별 알림을 복제하는 공간이 아니다. 다음처럼 행동이 필요한 항목만 모은다.

- 리뷰가 필요한 MR
- 실패한 Pipeline 또는 배포
- 마감 임박·마감 경과 일정
- 담당자 없는 중요 이슈
- 일정 동기화 실패 또는 연동 권한 만료

Mattermost는 알림 전달 채널이다. 일정·작업의 원본 저장소로 사용하지 않는다.

#### 프로젝트 시작 지원

- 팀장·팀원 식별 및 연결 상태 확인
- GitLab, Jira, Notion, Mattermost, Jenkins 연결 상태 표시
- 공통 링크, 배포 주소, 스크럼 템플릿, 일정 규칙 등록
- 허용된 범위에서 Mattermost 전용 채널·webhook 생성 지원
- 권한 부족 또는 인스턴스 정책으로 자동화가 실패하면 수동 설정 절차 제공

### 연동·동기화 원칙

#### 데이터 소유권

| 데이터 | 원본 도구 | 서비스 저장 범위 |
| --- | --- | --- |
| Jira 이슈 | Jira | ID, 상태, 담당자, 마감일, 원문 URL, 동기화 시각 |
| GitLab MR·Pipeline·배포 | GitLab | ID, 상태, 담당자, 요약 지표, 원문 URL, 동기화 시각 |
| 스크럼 문서·팀 문서 | Notion | Notion 페이지 ID, 제목, URL, 상태, 수정 시각 |
| 중앙 일정 | 서비스 | 전체 일정 정보, 담당자, 알림 설정, 외부 원본 참조 |
| Mattermost 메시지 | Mattermost | 메시지 원문을 기본 저장하지 않음; 발송 결과와 원문 링크만 보관 |

#### 동기화 규칙

- 한 일정은 하나의 원본만 가진다.
- 외부 원본의 변경은 webhook 우선, 주기적 재검증 보조 방식으로 중앙 화면을 갱신한다.
- 중앙 일정은 Notion 팀 일정 DB로 단방향 기록한다.
- 같은 데이터를 양방향으로 동시에 수정하지 않는다.
- 동기화 실패는 자동 덮어쓰기 대신 `확인 필요` 상태와 수동 재시도 기능으로 처리한다.
- Notion 문서 이동·휴지통 이동·권한 상실은 페이지 ID로 추적하고, 연결 상태만 변경한다.

### UX 원칙

- 첫 화면에서 팀원이 10초 안에 다음 행동을 정할 수 있어야 한다.
- 중요도·마감·실패 여부로 정렬하고, 정보량보다 행동 가능성을 우선한다.
- 도구의 상세 데이터는 원문 URL로 이동한다.
- 팀원 모두가 보는 화면은 팀 상태 중심, 개인 생산성 수치는 개인 화면 중심으로 둔다.
- 연동 설정은 ID·URL 수동 입력보다 연결 확인·목록 선택·자동 생성 흐름을 사용한다.
- 각 카드에는 원본 도구, 마지막 동기화 시각, 연결 상태를 표시한다.

### 우선순위

| 단계 | 포함 기능 |
| --- | --- |
| P0 | 팀/도구 연결, 오늘의 팀 홈, Jira 이슈·GitLab MR/Pipeline 읽기, 중앙 캘린더, 스크럼 템플릿·Notion 저장, 원문 링크 |
| P1 | Notion 일정 DB 선택·생성, Mattermost 알림, GitLab/Jira webhook, 기간 통계·시각화, 연동 상태·실패 처리 |
| P2 | GitLab 배포/릴리스 확장, Jenkins 연동, 선택형 Notion 외부 일정, Git graph 상세 보기, AI 문장 다듬기·주간 회고 |

### P0 성공 기준

- 팀원이 한 화면에서 자신의 이슈, 리뷰 대기 MR, 실패 Pipeline, 오늘 스크럼, 다가오는 일정을 확인한다.
- 스크럼 문서를 기본 또는 팀 템플릿으로 작성해 Notion에 저장한다.
- 팀장이 중앙에서 발표·제출·회의 일정을 만들고 담당자·리마인드를 설정한다.
- Jira와 GitLab의 각 항목은 정확한 원문 URL로 이동한다.
- 연동 권한 부족과 동기화 실패를 사용자가 알아볼 수 있다.

### 후속 기능

- AI 기반 스크럼 문장 다듬기와 주간 회고 요약
- Jira 이슈와 GitLab MR의 자동 연결 제안
- 팀 진행 위험 신호 고도화
- 중앙 일정과 Notion/Jira의 선택적 쓰기 동기화
- 배포·데모를 자동 모으는 릴리스 기록

## 최종 요약

제품의 중심은 도구 통합이 아니라 팀이 행동을 결정하는 운영 화면이다. P0는 작업 현황, 중앙 일정, 직접 작성하는 스크럼, Notion 저장으로 구성한다. 일정은 종류별 원본을 분리하고, Mattermost는 알림 채널로만 사용한다. 통계는 개인 경쟁이 아닌 팀의 진행·병목·기록을 시각화한다.
